El chip puede no latchear el byte correcto.

En este contexto, "ponerlo a nivel de kernel" no significa que el CPU cambie de modo ni que aparezca una fuente nueva de aleatoriedad. Significa que el recolector de jitter se ejecuta dentro del kernel, en espacio kernel, en vez de como proceso de usuario. El algoritmo base sigue siendo el mismo: tomar marcas de tiempo de alta resolución, medir las variaciones entre lecturas consecutivas, mezclar esos deltas con un pool interno y aplicar tests de salud. En el PDF de Müller, para Linux en kernel space la obtención del tiempo se hace con random_get_entropy(), y si eso devuelve 0 usa __getnstimeofday() como fallback al mejor clocksource disponible. El propio repositorio sigue describiendo la librería como una fuente de ruido basada en el jitter temporal del CPU y no dependiente del sistema operativo


tu proceso o hilo entra en Jitterentropy,

Jitterentropy lee el TSC,

ejecuta su bucle interno, hash, accesos a memoria, tests, etc.,

vuelve a leer el TSC,

calcula el delta.

Ese delta no pertenece a ningún proceso concreto. Refleja todo lo que afectó a ese intervalo de ejecución: estado interno de la CPU, cachés, pipeline, branch predictor, TLB, wait states de memoria, cambios de frecuencia, interrupciones, migraciones del scheduler y carga de otras aplicaciones. El propio documento enumera precisamente esos factores como fuentes de variación temporal. 

CPU-Jitter-NPTRNG

Por eso la forma correcta de decirlo no es "lee el TSC de otros procesos", sino:

lee el contador global/de CPU mientras su propio código corre, y en ese tiempo quedan mezclados los efectos del resto del sistema.

De hecho, Müller también muestra que en user space, si hay muchas aplicaciones ejecutándose, incluido X11, el jitter aumenta respecto a un sistema casi quieto. O sea, otras cargas del sistema influyen en la medición, pero no porque el RNG lea "sus procesos", sino porque alteran cuánto tarda en ejecutarse el código entre dos rdtsc. 

CPU-Jitter-NPTRNG

Y aún más: en las pruebas bare metal sin sistema operativo, sin scheduling ni context switches, sigue habiendo jitter medible. Eso refuerza que la fuente principal no es "leer procesos", sino la propia microarquitectura de la CPU y la memoria.



A nivel de user space, en la implementación actual de la librería no se “espera” a otros procesos para crear jitter. En x86/x86_64, jent_get_nstime() lee directamente el contador con la instrucción rdtsc. En AArch64 usa el contador del sistema (cntvct_el0), y en otras plataformas puede caer a clock_gettime(CLOCK_REALTIME) o a mecanismos equivalentes como mach_absolute_time(). De hecho, el propio código comenta que en algunos casos usa CLOCK_REALTIME porque de vez en cuando puede recibir algo de variación extra por ajustes de NTP, aunque no depende de eso.

El TSC es el Time Stamp Counter. En la documentación del kernel de Linux para x86 se describe como un contador temporal de 64 bits del procesador, legible con RDTSC o RDTSCP, que se usa como medida de tiempo. También advierte que puede ser problemático como fuente temporal porque es local a la CPU y no siempre está perfectamente sincronizado entre núcleos o sockets. En Linux/x86, random_get_entropy() devuelve rdtsc() cuando el TSC está disponible; si no, usa un fallback.

si fuerzo a que entre en fallback? aunque el clocksource sea muy preciso con muchos decimales no es determinista?
que el clocksource tenga muchos decimales no significa que deje de ser determinista. Resolución y entropía no son lo mismo. Puedes tener una API que te entregue nanosegundos y, aun así, que el comportamiento sea muy predecible si el contador subyacente es estable, emulado o si el entorno ejecuta siempre casi igual. El propio proyecto avisa de que un hipervisor puede emular rdtsc y degradar la entropía, y la librería hace tests de arranque y de salud para comprobar si el sistema realmente ofrece las propiedades necesarias para medir y recolectar entropía; si no, bloquea la salida o considera el sistema no seguro para ese uso.

Si fuerzas el fallback, lo que cambias no es solo "la precisión del reloj", sino también la ruta de código para leerlo. En kernel, si no hay random_get_entropy() utilizable, acaba en ktime_get_ns(), y eso a su vez usa la infraestructura de timekeeping del kernel, que termina leyendo el clocksource activo mediante clock->read(clock). En otras palabras, ya no mides un rdtsc casi directo, sino un camino más largo de timekeeping. Eso puede introducir latencias y variaciones propias del software de timekeeping, pero también puede ser peor si el clocksource activo es basto, sintético o demasiado regular. No se puede asumir a priori que será mejor: hay que medirlo y dejar que los health tests lo validen.

La SHA-3 dentro de Jitterentropy no es relleno. En el diseño evaluado por Müller, la operación SHA-3 tiene dos funciones reales. La primera es servir como carga de trabajo cuya duración se mide: el documento dice explícitamente que la operación SHA-3 "is considered as a noise source" porque su ejecución contiene variaciones temporales, y que el corazón del generador mide la ejecución de la parte de hash y la de accesos a memoria como una única fuente de ruido. La segunda es actuar como condicionador/mezclador: los deltas de tiempo se van inyectando en un pool de entropía mediante SHA-3, y el propio documento dice que la salida de SHA3-256 es la salida del Jitter RNG.

a entropía pasa por un hash. Más exactamente, no se entregan los deltas crudos del temporizador; se mezclan en un estado hash y lo que sale al caller es la salida condicionada de ese estado. El PDF también lo trata como un "vetted conditioning component" bajo SP800-90B. En la integración actual del kernel Linux, además, la constante visible sigue siendo JENT_CONDITIONING_HASH "sha3-256", y el bucle jent_hash_time() existe precisamente para ejecutar una operación sobre la que hacer la medición temporal.

#!/usr/bin/env bash
set -uo pipefail

SCRIPT_DIR="$(CDPATH= cd -- "$(dirname -- "$0")" && pwd)"
PROCESSDATA_SH="${PROCESSDATA_SH:-$SCRIPT_DIR/processdata.sh}"
DATA_ROOT_INPUT="${1:-../results-measurements/concurrency}"
RESULTS_ROOT_INPUT="${2:-../results-analysis-runtime/concurrency}"
DATA_ROOT="$(realpath -m "$DATA_ROOT_INPUT")"
RESULTS_ROOT="$(realpath -m "$RESULTS_ROOT_INPUT")"
OSR="${OSR:-3}"
DETAIL_FILE="$RESULTS_ROOT/concurrency_detail.tsv"
SUMMARY_FILE="$RESULTS_ROOT/concurrency_summary.tsv"
FAIL_FILE="$RESULTS_ROOT/concurrency_failures.tsv"

mkdir -p "$RESULTS_ROOT"
THRESHOLD="$(awk -v osr="$OSR" 'BEGIN { printf "%.12f", 1/osr }')"

extract_metric() {
  local metric_file="$1"
  awk -F': *' '
    /^H_original:/ { h_original=$2 }
    /^H_bitstring:/ { h_bitstring=$2 }
    /^min\(H_original, 8 X H_bitstring\):/ { minval=$2 }
    END {
      printf "%s\t%s\t%s\n", h_original, h_bitstring, minval
    }
  ' "$metric_file"
}

count_health_flags() {
  local log_file="$1"
  if [ ! -f "$log_file" ]; then
    printf '0\t0\t0\n'
    return 0
  fi
  awk '
    /RCT/ { rct++ }
    /APT/ { apt++ }
    /Lag/ { lag++ }
    END {
      printf "%d\t%d\t%d\n", rct+0, apt+0, lag+0
    }
  ' "$log_file"
}

median_from_file() {
  local file="$1"
  sort -g "$file" | awk '
    { a[NR]=$1 }
    END {
      if (NR == 0) exit 1
      if (NR % 2 == 1) {
        printf "%.12f\n", a[(NR+1)/2]
      } else {
        printf "%.12f\n", (a[NR/2] + a[NR/2 + 1]) / 2
      }
    }
  '
}

run_processdata() {
  local sample_dir="$1"
  local result_dir="$2"
  (
    cd "$SCRIPT_DIR"
    "$PROCESSDATA_SH" "$sample_dir" "$result_dir" >/dev/null 2>&1
  )
}

printf 'parallel_instances\tinstance\tH_original\tH_bitstring\tH_min\tbelow_threshold\tRCT\tAPT\tLag\tsample_dir\tresults_dir\n' > "$DETAIL_FILE"
printf 'parallel_instances\tinstances_ok\tH_min_mean\tH_min_median\tH_min_min\tbelow_threshold_count\tRCT_total\tAPT_total\tLag_total\tthreshold\n' > "$SUMMARY_FILE"
printf 'parallel_instances\tinstance\treason\tsample_dir\tresults_dir\n' > "$FAIL_FILE"

shopt -s nullglob
for p_dir in "$DATA_ROOT"/p_*; do
  case_name="$(basename "$p_dir")"
  p_value="${case_name#p_}"
  case_results_root="$RESULTS_ROOT/$case_name"
  mkdir -p "$case_results_root"

  tmp_vals="$(mktemp)"

  ok_count=0
  rct_total=0
  apt_total=0
  lag_total=0

  for inst_dir in "$p_dir"/inst_*; do
    inst_name="$(basename "$inst_dir")"
    inst_results_dir="$case_results_root/$inst_name"
    mkdir -p "$inst_results_dir"

    if ! run_processdata "$inst_dir" "$inst_results_dir"; then
      printf '%s\t%s\tprocessdata_failed\t%s\t%s\n' \
        "$p_value" "$inst_name" "$inst_dir" "$inst_results_dir" >> "$FAIL_FILE"
      continue
    fi

    metric_file="$inst_results_dir/jent-raw-noise-0001.minentropy_FF_8bits.txt"
    if [ ! -f "$metric_file" ]; then
      printf '%s\t%s\tmissing_metric_file\t%s\t%s\n' \
        "$p_value" "$inst_name" "$inst_dir" "$inst_results_dir" >> "$FAIL_FILE"
      continue
    fi

    IFS=$'\t' read -r h_original h_bitstring h_min < <(extract_metric "$metric_file")

    if [ -z "$h_min" ]; then
      printf '%s\t%s\tempty_metric\t%s\t%s\n' \
        "$p_value" "$inst_name" "$inst_dir" "$inst_results_dir" >> "$FAIL_FILE"
      continue
    fi

    IFS=$'\t' read -r rct apt lag < <(count_health_flags "$p_dir/${inst_name}.log")

    below="$(awk -v v="$h_min" -v t="$THRESHOLD" 'BEGIN { print (v+0 < t+0) ? 1 : 0 }')"
    printf '%s\n' "$h_min" >> "$tmp_vals"
    ok_count=$(( ok_count + 1 ))
    rct_total=$(( rct_total + rct ))
    apt_total=$(( apt_total + apt ))
    lag_total=$(( lag_total + lag ))

    printf '%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\n' \
      "$p_value" "$inst_name" "$h_original" "$h_bitstring" "$h_min" "$below" \
      "$rct" "$apt" "$lag" "$inst_dir" "$inst_results_dir" >> "$DETAIL_FILE"
  done

  if [ "$ok_count" -eq 0 ]; then
    rm -f "$tmp_vals"
    continue
  fi

  mean_hmin="$(awk '{ s += $1 } END { if (NR>0) printf "%.12f", s / NR }' "$tmp_vals")"
  median_hmin="$(median_from_file "$tmp_vals")"
  min_hmin="$(sort -g "$tmp_vals" | head -n1)"
  below_count="$(awk -v t="$THRESHOLD" '$1+0 < t+0 { c++ } END { print c+0 }' "$tmp_vals")"

  printf '%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\t%s\n' \
    "$p_value" "$ok_count" "$mean_hmin" "$median_hmin" "$min_hmin" \
    "$below_count" "$rct_total" "$apt_total" "$lag_total" "$THRESHOLD" >> "$SUMMARY_FILE"

  rm -f "$tmp_vals"
done

echo "Wrote $DETAIL_FILE"
echo "Wrote $SUMMARY_FILE"
echo "Wrote $FAIL_FILE"
